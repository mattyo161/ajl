# Data files

Every file or directory ajl reads or writes, its format, and how it relates
to the environment variables in [environment.md](environment.md) that
control it. Nothing here is required — ajl works with none of these present
(just AWS credentials); they exist to make repeat runs cheaper, auditable,
or safer to pipe around.

## The result cache — `~/.cache/ajl/` (or `AJL_CACHE_DIR`)

Written by `cache.py` when `--cache TTL` (or `AJL_CACHE`) is set. One
invocation's cache entry is three files sharing a 32-hex-char key:

```
<key>.meta.json          # always
<key>.jsonl.gz           # unencrypted body
<key>.jsonl.gz.age       # age-encrypted body, instead of .jsonl.gz — never both
```

**The key** (`ResultCache.key`) is `sha256(json)[:32]` over: ajl version,
service, operation, remaining CLI tokens, resolved profile/region, every
output-affecting flag (`--jq`, `--stamp-session`, `--describe`,
`--fetch-tags`, `--no-parse`, `--no-paginate`, `--max-items`), the fan-out
flags (`--all`/`--all-profiles`/`--all-regions`/`--profiles`/`--regions`),
and a sha256 of the full `--params-json` input if any. Two invocations
produce the same key if and only if they'd produce the same output — which
is also why every flag that changes what gets *emitted* has to be added to
that material by hand (see DESIGN.md's "Fix: `--describe` wasn't part of
the cache key material" for what happens when one is missed).

**`<key>.meta.json`** — a small JSON object, not gzipped:

```json
{"key": "...", "created": 1752345600.0, "expires": 1752950400.0,
 "argv": ["ecs", "list-clusters", "--all-regions"],
 "duration": 4.2, "lines": 37, "encrypted": true}
```

`created`/`expires` are epoch seconds (`expires` drives `--rm-after`
cleanup — default `7d`, `AJL_CACHE_RM_AFTER`). `ajl cache ls` reads exactly
this file per entry (see [commands/cache.md](commands/cache.md)).

**`<key>.jsonl.gz` / `<key>.jsonl.gz.age`** — the gzipped JSONL body,
optionally age-encrypted on top (`AJL_AGE_*` — see environment.md). Which
one exists tells you whether that entry is encrypted; `commit()` never
leaves both — writing the encrypted form deletes a stale unencrypted one
for the same key, and vice versa.

**Lifecycle**: only an exit-0 run commits an entry (`open_writer`/
`CacheWriter.commit`) — a killed or failed run's partial output is never
cached (`.abort()` deletes the temp file instead). Every cache-enabled
invocation opportunistically sweeps expired entries first (`sweep()`),
so there's no separate cron/reaper process — the cache tidies itself as a
side effect of being used. `ajl cache clear` (no args) does the same sweep
on demand; `ajl cache clear --all` removes everything regardless of expiry.

## State: the learn log — `~/.local/state/ajl/learn.jsonl` (or `AJL_LEARN_FILE`)

Written by `learn.py` when `--learn`/`AJL_LEARN=1` is active — **one JSONL
line per ajl invocation** (not per API call; see api-log below for that).
Every line has a `Ts` (ISO-8601 UTC, second precision) plus:

```json
{"Ts": "2026-07-16T22:10:03+00:00", "Command": "ajl ecs list-clusters --all-regions",
 "Service": "ecs", "Operation": "list-clusters", "Profile": "nri-customer",
 "Region": "us-east-1", "Cache": "miss", "AwsEquivalent": "aws ecs list-clusters ...",
 "ExitCode": 0, "DurationS": 1.24}
```

`Cache` is `"off"` (no `--cache`), `"miss"`, or `"hit"` (in which case
`SavedS` — the original run's duration — is also present, so you can see
what a hit *saved* you). For `s3 scan`/`s3 list`, the report's run stats
(tasks, requests, bytes, a small sample of the actual keyspace slices
listed) get merged in too. Append-only, never rotated by ajl itself.

## State: the per-API-call telemetry log — `~/.local/state/ajl/apilog.jsonl` (or `AJL_APILOG_FILE`)

Written by `apilog.py` when `--api-log`/`AJL_APILOG=1` is active — **one
JSONL line per underlying botocore call**, so a single paginated or
fanned-out ajl invocation can produce many lines here for the one line it
produces in the learn log:

```json
{"Ts": "2026-07-17T04:00:36.098+00:00", "Service": "s3", "Operation": "ListBuckets",
 "Profile": "", "Region": "us-east-1", "DurationS": 0.741, "HttpStatus": 200,
 "Attempts": 1, "Items": 28, "Outcome": "success"}
```

On failure, `HttpStatus`/`Items` are `null` and an `ErrorCode` field is
added instead. **Known gap** (see `apilog.py`'s module docstring and
DESIGN.md): each boto3 client gets its own independent copy of the event
emitter it's hooked into, so this never sees credential-resolution
sub-clients (SSO `GetRoleCredentials`, an `AssumeRole` fetcher) — only calls
made through clients ajl itself builds via `Runner.client()`. A gap in this
log on an otherwise-slow request points at credential resolution rather
than at the call ajl made.

## Model files — `src/ajl/models/*.json` (packaged, or `AJL_MODELS_DIR`)

One file per AWS service, generated by `tools/generate-model.py` from the
boto3/botocore service definitions and then hand-curated by
`tools/apply-resource-configs.py` (see [request-flow.md](request-flow.md)
for how these drive shaping at runtime — this section is about the file
itself). **Never hand-edit these** — regeneration wipes anything not
written by `apply-resource-configs.py`; edit the curation script and
re-run it. Shape:

```json
{
  "operations": {
    "ListClusters": {
      "input": {"members": {"maxResults": {"type": "integer"}, ...}, "markers": [...]},
      "output": {
        "members": {"clusterArns": {"type": "list"}, ...},
        "markers": [...],
        "resources": [{"path": ["clusterArns"], "type": "ecs:cluster",
                        "arn": "clusterArn", "scalar_as": "clusterArn"}],
        "jq": "...",
        "describe": {"operation": "DescribeClusters", "id_field": "Id",
                      "param": "clusters", "kind": "array", "batch_size": 100}
      }
    }
  }
}
```

`output.jq`, `output.resources`, and `output.describe` are independent —
an operation can have any combination (or none, falling back to the
heuristic path). `AJL_MODELS_DIR` points ajl at a different directory
entirely (typically `src/ajl/models` itself during development, so curation
edits are visible without reinstalling — every example in these docs that
sets it is doing exactly that).

## `~/.aws/config` (or `AWS_CONFIG_FILE`) — read, never written

`fanout.py`'s `resolve_profiles()` parses this with `configparser` to
discover profile names for `--all-profiles`/`--all` (unless `AJL_PROFILES`
is set, which replaces this as the source list — see environment.md for
the dedup relationship between the two). ajl never writes to this file.

## age identity file (optional) — wherever `AJL_AGE_IDENTITY` points

If `AJL_AGE_IDENTITY` doesn't start with `AGE-SECRET-KEY-`, `cache.py`
treats it as a path and reads the first line starting with
`AGE-SECRET-KEY-` out of it. `ajl cache keygen` prints a ready-to-save
identity plus its public key to stdout (see
[commands/cache.md](commands/cache.md)) but does not write a file itself —
redirecting that output, or exporting the printed key directly, is up to
you.

## Sealed values — not a file, but worth noting here

`AJLSEC:1:<base64>` envelopes (`seal.py`) live *inline* in JSONL output —
they're a value shape, not a separate file. They travel through pipes,
`--params-json` round-trips, and the result cache untouched (the cache
doesn't know or care that a value is sealed) and are only ever unsealed by
`ajl decrypt` or `ssm get --decrypt` reading the same `AJL_AGE_*`
configuration. Mentioned here because they're the one persistent-ish
"data format" that crosses every other mechanism on this page without
having a file of its own.
